# 🗺️ План модернізації SSSK: Етап P1 (Автономність та Інфраструктура)

## 🎯 Мета (Goal)
Створити нову автономну інфраструктуру PWA: модульну структуру коду, локальний шар даних, базові сервіси, міст до парсера та архітектурну основу для майбутньої синхронізації з SQL-бекендом і системою пуш-повідомлень.

## 🚫 Поза межами етапу (Out of Scope for P1)
*   Supabase Auth / Google Sign-In / Remote DB Sync.
*   Реальна push-система (Firebase/Web-Push) — готуємо лише інфраструктуру під неї.

---

## 🛠️ Етапи реалізації

### 1. Build & App Foundation (Vite)
*   Перенесення проєкту на **Vite** як нову build-основу.
*   **PWA Lifecycle**: Налаштування логіки виявлення оновлень ("Доступна нова версія. Оновити?").
*   Реорганізація структури: перенесення коду з `pwa/` у модульну структуру `/src`.
*   Налаштування alias-и, scripts та build/preview pipeline.

### 2. Core Structure (Чиста Архітектура)
*   Розділення Presentation, Application, Domain, Infrastructure.
*   **UI Notification Service**: Створення базового сервісу для виводу сповіщень (тостів) користувачу (наприклад, "Дані збережено").

### 3. Local Data Foundation (База на борту)
*   Створення **DataManager** (єдиний шар доступу до даних).
*   **Schema Versioning**: Кожна JSON-база має поле `version` для майбутніх міграцій на SQL.
*   Впровадження еталонних JSON-схем:
    - `users` (профіль, тема, налаштування);
    - `cards` (дані про підчерги);
    - `push_inventory` (токен, статус тогла "на завтра").
*   Забезпечення, щоб форми кабінету працювали з цими структурами.

### 4. Parser Bridge (Місток)
*   Read-only моніторинг локальних файлів `pwa/data/today.json` та `tomorrow.json`.
*   Виявлення змін (новий графік) та активація кнопки "Графік на завтра".
*   Підготовка подій для майбутньої push-логіки.

### 5. Cabinet Logic (3-карткова модель)
*   Перебудова кабінету на 3-карткову модель (2 для сьогодні + 1 для завтра).
*   Форма профілю: зміна username/location з валідацією.
*   **Theme Persistence**: Збереження теми (Dark/Light) у профілі користувача.
*   Збереження змін за кнопкою "Зберегти" (без автозбереження).

### 6. Admin Console v2
*   Локальна адмін-панель для керування тестовими даними.
*   Редагування тестових клієнтів вручну.
*   Механізм `blacklist.json` через GitHub для віддаленого блокування.

---

## ✅ Критерії успіху
1. Проєкт перейшов на нову Vite-структуру.
2. Локальний DataManager працює стабільно з версійними схемами.
3. Кабінет зберігає дані у структурованому вигляді.
4. Parser bridge коректно читає дані від парсера.
5. Кодова архітектура готова до підключення Supabase на етапі P2.
